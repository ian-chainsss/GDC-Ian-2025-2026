PostgreSQL, voorbij wat elke SQL-database kan

Je kent MariaDB al: tabellen, indexen, transacties, joins — dat fundament sla ik over. Dit document zoomt in op wat PostgreSQL extra doet: datatypes die verder gaan dan VARCHAR en JSON, een ander intern model voor transacties, indexen voor gevallen waar B-tree tekortschiet, en een uitbreidingsmechanisme waarmee de database zelf meegroeit met wat je nodig hebt. Overal waar het relevant is, leg ik de brug naar een FastAPI-backend met Redis ernaast — zonder al naar het eindproject toe te werken.

01

Filosofie: waarom Postgres anders aanvoelt

MariaDB en PostgreSQL lossen "een SQL-database bouwen" op met een andere grondgedachte, en dat verklaart bijna elk verschil verderop in dit document.

MariaDB is gebouwd rond vervangbare storage-engines: InnoDB voor transacties, Aria voor lichte workloads, ColumnStore voor analytics, MyRocks voor schrijfzware workloads — je kiest de motor per tabel. PostgreSQL doet het omgekeerde: er is precies één storage-engine, maar bijna alles daarboven is uitbreidbaar. Je kan zelf nieuwe datatypes registreren, nieuwe indexmethodes toevoegen, functies schrijven in Python of PL/pgSQL, en zelfs externe databases laten aanvoelen als een lokale tabel via foreign data wrappers. Die uitbreidbaarheid is precies waarom PostgreSQL het "Perl van de databases" wordt genoemd: het is niet één product, het is een platform waar features als JSONB, PostGIS en TimescaleDB bovenop zijn gebouwd zonder de kernbroncode aan te raken.

Een tweede verschil: PostgreSQL volgt de SQL-standaard opvallend strikt en implementeert features vaak het eerst of het volledigst — window functions, recursieve CTE's, echte SERIALIZABLE isolatie, MERGE. MariaDB (en MySQL) hebben historisch pragmatischer gekozen: sneller shippen, soms met een eigen syntax-dialect. Geen van beide is "fout" — het is een andere prioriteit.

🗓️

De actuele stabiele release is PostgreSQL 18 (september 2025); versie 19 is sinds juni 2026 in bèta. Waar een feature pas sinds een specifieke versie bestaat, vermeld ik dat expliciet — dat is relevant zodra je een concrete serverversie kiest voor een project.

▸Wat dit document wél en niet doet

Ik ga ervan uit dat je vertrouwd bent met gewone SQL: SELECT, JOIN, transacties, primaire/vreemde sleutels, gewone B-tree-indexen. Elk hoofdstuk hieronder behandelt iets wat in een standaard-SQL-database ofwel niet bestaat, ofwel merkbaar zwakker is uitgewerkt. Het einddoel — een cashless betaalsysteem — komt hier bewust niet aan bod; de voorbeelden zijn generiek zodat je het concept kan loskoppelen van die ene toepassing.

02

Geavanceerde datatypes

Dit is waarschijnlijk het meest zichtbare verschil in je dagelijkse werk: PostgreSQL heeft een rijkere kolom-toolbox, en die kolommen zijn ook indexeerbaar — niet enkel handig om in op te slaan.

▸JSONB: binaire JSON, niet tekst-met-een-label

json in PostgreSQL slaat de exacte tekst op (inclusief witruimte en dubbele keys) en parseert die opnieuw bij elke bewerking. jsonb ontleedt het document bij het invoegen tot een binair, genest formaat: iets trager om te schrijven, veel sneller om te doorzoeken, en — cruciaal — indexeerbaar.

psql
CREATE TABLE orders (
    id       uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    customer_id integer NOT NULL,
    payload  jsonb NOT NULL
);

INSERT INTO orders (customer_id, payload) VALUES
    (42, '{"items": [{"sku": "A1", "qty": 2}],
           "status": "paid",
           "meta": {"channel": "web"}}');

-- Padnavigatie
SELECT payload -> 'items'          FROM orders;  -- jsonb
SELECT payload ->> 'status'        FROM orders;  -- text
SELECT payload #> '{meta,channel}' FROM orders;   -- pad, als jsonb

-- "Bevat dit document dit fragment?" — de kernquery voor JSONB
SELECT * FROM orders WHERE payload @> '{"status": "paid"}';

-- Index die @>, ? en verwante operatoren versnelt
CREATE INDEX idx_orders_payload ON orders USING GIN (payload);
vs MariaDB

MariaDB's JSON-type is een alias voor LONGTEXT met een JSON_VALID()-check erbovenop — geen binair formaat, geen native index op een JSON-pad. De gangbare omweg is een virtuele gegenereerde kolom met een gewone (functionele) index erop. In MySQL zelf ligt dat anders: dat heeft sinds 5.7 wél een echt binair JSON-type. "MariaDB-compatibel" en "MySQL-compatibel" zijn dus niet hetzelfde zodra JSON in het spel is.

▸Native arrays

Elke kolomtype kan een array van dat type zijn — geen aparte junctietabel nodig voor eenvoudige lijsten zoals tags.

psql
CREATE TABLE products (
    id   serial PRIMARY KEY,
    name text,
    tags text[]
);

INSERT INTO products (name, tags) VALUES ('Arduino Uno', ARRAY['electronics','maker','3v3']);

SELECT * FROM products WHERE 'maker' = ANY(tags);        -- bevat element
SELECT * FROM products WHERE tags @> ARRAY['maker'];      -- containment
SELECT unnest(tags) FROM products;                        -- terug naar rijen

CREATE INDEX idx_products_tags ON products USING GIN (tags);

▸UUID: gen_random_uuid() en het nieuwe uuidv7()

uuid is een eersteklas 16-byte type, geen CHAR(36)-string. Sinds PostgreSQL 13 zit gen_random_uuid() ingebouwd in de core (voorheen had je de pgcrypto-extensie nodig). Sinds PostgreSQL 18 is er ook uuidv7(): een tijd-gesorteerde UUID, waarbij de eerste bytes een timestamp coderen.

FunctieTypeSorteringWanneer kiezen
gen_random_uuid()UUIDv4 (random)willekeuriggeen aanname over volgorde nodig
uuidv7()UUIDv7 (tijd + random)chronologischprimaire sleutel op een grote, snel-groeiende tabel — betere B-tree-lokaliteit, minder pagina-fragmentatie bij inserts
📎

Waarom dat lokaliteitsverschil er is: een volledig willekeurige UUIDv4-sleutel spreidt inserts over de hele B-tree-index, wat leidt tot willekeurige schijftoegang en een index die trager groeit in cache-vriendelijkheid. UUIDv7 gedraagt zich in die zin meer als een oplopende bigserial, terwijl je toch de voordelen van een globaal unieke, niet-voorspelbare sleutel behoudt.

▸ENUM, ranges en samengestelde types

psql
-- ENUM: gevalideerd door de database, niet enkel door je applicatiecode
CREATE TYPE order_status AS ENUM ('pending', 'paid', 'refunded', 'cancelled');
ALTER TABLE orders ADD COLUMN status order_status DEFAULT 'pending';

-- Range types: één kolom voor een interval, met eigen operatoren
CREATE TABLE price_history (
    product_id integer,
    price      numeric,
    valid_for  daterange
);
SELECT * FROM price_history WHERE valid_for @> CURRENT_DATE;      -- bevat dit punt?
SELECT * FROM price_history WHERE valid_for && daterange('2026-08-01','2026-09-01');  -- overlapt?

-- Composite type: een eigen "struct"
CREATE TYPE address AS (street text, city text, postcode text);

Die daterange/tstzrange-types zijn hier bewust kort geïntroduceerd — in hoofdstuk 06 blijkt waarom ze de basis vormen van een van de meest onderschatte PostgreSQL-features: constraints die overlap zelf tegenhouden.

▸Netwerktypes & full-text search

Twee kleinere maar handige categorieën: native netwerktypes (relevant gezien je netwerkachtergrond) en ingebouwde tekstzoek-types.

psql
-- inet/cidr: validatie én subnet-operatoren, geen VARCHAR met regex
CREATE TABLE access_log (id serial, client inet, allowed_from cidr);
SELECT * FROM access_log WHERE client <<= allowed_from;   -- zit client-IP in dit subnet?

-- Full-text search: tsvector/tsquery + GIN-index, geen LIKE '%...%'
ALTER TABLE products ADD COLUMN search tsvector
    GENERATED ALWAYS AS (to_tsvector('dutch', name)) STORED;
CREATE INDEX idx_products_search ON products USING GIN (search);
SELECT * FROM products WHERE search @@ plainto_tsquery('dutch', 'arduino maker');
Dieper: hstore, nog steeds tegenkomen in oudere code

Vóór jsonb bestond (PostgreSQL < 9.4) was hstore de manier om key-value-paren in één kolom te stoppen — een platte tekst-naar-tekst map, geen geneste structuur. Je vindt het nog in legacy-schema's en het is nog steeds sneller voor puur platte key-value-opslag, maar voor nieuw werk is jsonb vrijwel altijd de betere keuze: geneste structuren, rijkere operatoren, en dezelfde GIN-indexeerbaarheid.

03

MVCC: hoe een UPDATE écht werkt

Zowel InnoDB als PostgreSQL gebruiken multi-version concurrency control zodat lezers nooit blokkeren op schrijvers. Maar de manier waarop ze oude versies bewaren, verschilt fundamenteel — en dat verklaart waarom PostgreSQL VACUUM nodig heeft en InnoDB niet.

vs MariaDB (InnoDB)

InnoDB werkt in-place: de rij wordt direct aangepast, en de oude waarde gaat naar een apart undo log (rollback-segment) zodat lopende transacties de oude versie nog kunnen zien. PostgreSQL past de rij nooit in-place aan: een UPDATE schrijft een volledig nieuwe rijversie (tuple) ergens in de tabel-heap, en markeert de oude als dood. De oude versie blijft fysiek liggen tot VACUUM ze opruimt.

Elke tuple heeft twee onzichtbare systeemkolommen: xmin (transactie-ID die de rij aanmaakte) en xmax (transactie-ID die de rij "verwijderde" — bij een UPDATE is dat de oude versie). Een lezende transactie ziet enkel tuples waarvan xmin voor haar snapshot gecommit is én xmax nog niet — dat is de kern van MVCC in PostgreSQL, zonder locks.

Dit is de reden waarom een tabel met veel UPDATEs en een te trage autovacuum kan "opzwellen" (bloat): de tabel op schijf wordt groter dan de data die ze bevat, omdat dode tuples blijven liggen. In productie is autovacuum-tuning — niet het uitschakelen ervan — een van de eerste dingen om te controleren bij een schrijfzware tabel.

▸Isolatieniveaus: PostgreSQL doet écht SERIALIZABLE

Het standaardniveau is READ COMMITTED, net als bij InnoDB. Het verschil zit in het strengste niveau: PostgreSQL's SERIALIZABLE is Serializable Snapshot Isolation (sinds 9.1) — de database detecteert gevaarlijke concurrency-patronen tussen transacties (niet enkel rij-conflicten) en laat er één falen met een serialization_failure, in plaats van stil een inconsistent resultaat toe te staan. Dat is een garantie die veel systemen enkel op papier bieden.

04

Indexen voorbij B-tree

B-tree ken je al — gelijkheid en bereik, dat werkt overal hetzelfde. PostgreSQL breidt dat uit met indexmethodes voor data die geen natuurlijke lineaire volgorde heeft.

TypeGoed voorVoorbeeld
B-treegelijkheid, bereik, sortering (het standaardgeval)WHERE email = ...
GIN"bevat dit element" in samengestelde waardenjsonb, arrays, full-text
GiSTgeometrie, ranges, nearest-neighbor, exclusion constraintsoverlappende periodes, locaties
BRINenorme, fysiek gesorteerde tabellen (append-only)logtabellen op created_at
Hashpure gelijkheid, iets compacter dan B-treezelden de eerste keuze; B-tree kan bijna alles net zo goed

Twee kruisingen daarmee die vaak over het hoofd worden gezien:

  • Partiële index — indexeer enkel de rijen die je effectief opzoekt: CREATE INDEX ON orders (id) WHERE status = 'pending';. Kleiner, sneller te onderhouden, en de planner gebruikt hem automatisch zodra je query dezelfde WHERE bevat.
  • Expressie-index — indexeer het resultaat van een functie, niet de ruwe kolom: CREATE INDEX ON users (lower(email)); maakt WHERE lower(email) = 'ian@x.be' een index-scan in plaats van een sequential scan.
  • Covering index (INCLUDE) — voeg extra kolommen toe die niet meetellen voor de sortering maar wel meegeleverd worden, zodat een query volledig index-only beantwoord kan worden zonder de tabel-heap te raken.

Gewone gelijkheid op een scalaire kolom → B-tree, de standaardkeuze in vrijwel elk geval.

2. WHERE payload @> '{"status":"paid"}' op een jsonb-kolom.

Containment-checks op geneste/samengestelde waarden (jsonb, arrays, full-text) → GIN.

3. Een logtabel van 500 miljoen rijen, ingevoegd in tijdsvolgorde, query op created_at > now() - interval '1 day'.

Data die fysiek al gesorteerd op schijf staat → BRIN: een fractie van de grootte van B-tree, want het slaat enkel min/max per blok op.

4. WHERE status = 'pending', maar 99% van de rijen heeft status = 'completed'.

Als je enkel de zeldzame waarde ooit opzoekt, indexeer dan enkel die rijen — kleiner en efficiënter dan de hele kolom indexeren.

5. ORDER BY location <-> point(4.35, 50.85) LIMIT 5 — dichtstbijzijnde 5 punten.

Nearest-neighbor en geometrische/range-operatoren → GiST, gebouwd voor "hoe dicht/hoezeer overlapt dit".
05

Krachtige querytechnieken

Een reeks SQL-features die de standaard voorschrijft, maar die PostgreSQL vollediger of eerder implementeert dan de meeste andere databases.

▸UPSERT: ON CONFLICT

INSERT INTO stock (sku, qty) VALUES ('A1', 5)
ON DUPLICATE KEY UPDATE qty = qty + VALUES(qty);
-- werkt enkel via de bestaande unieke/primaire sleutel,
-- geen controle over WHICH conflict getriggerd werd
INSERT INTO stock (sku, qty) VALUES ('A1', 5)
ON CONFLICT (sku) DO UPDATE
    SET qty = stock.qty + EXCLUDED.qty
    WHERE stock.qty < 1000
RETURNING *;
-- je kiest expliciet welke constraint het conflict afvangt,
-- kan een voorwaarde toevoegen, en direct het resultaat terugkrijgen

▸Recursieve CTE's

Hiërarchische data (org-structuren, categorieën, boomstructuren) zonder applicatiecode die recursief queryt:

psql
WITH RECURSIVE org_chart AS (
    SELECT id, name, manager_id, 1 AS depth
    FROM employees WHERE manager_id IS NULL
    UNION ALL
    SELECT e.id, e.name, e.manager_id, oc.depth + 1
    FROM employees e
    JOIN org_chart oc ON e.manager_id = oc.id
)
SELECT * FROM org_chart ORDER BY depth;

▸Window functions

Berekeningen over een venster van rijen, zonder de resultaatset zelf te reduceren zoals GROUP BY doet:

psql
SELECT
    customer_id,
    created_at,
    amount,
    SUM(amount) OVER (PARTITION BY customer_id ORDER BY created_at) AS lopend_totaal,
    RANK()      OVER (PARTITION BY customer_id ORDER BY amount DESC) AS rang
FROM orders;

▸RETURNING — nu ook met OLD en NEW (PG 18)

RETURNING geeft gewijzigde rijen direct terug zonder een aparte SELECT — onmisbaar in een API die na een INSERT/UPDATE meteen het volledige object wil serialiseren. Sinds PostgreSQL 18 kan je ook OLD en NEW apart opvragen bij een UPDATE:

psql
UPDATE orders SET status = 'paid'
WHERE id = 1
RETURNING OLD.status AS vorige_status, NEW.status AS nieuwe_status;
Dieper: LATERAL joins en FILTER

LATERAL laat een subquery rechts van een JOIN verwijzen naar kolommen van de rij links ervan — ideaal voor "top-N per groep": bijvoorbeeld de 3 meest recente bestellingen per klant, zonder een window function en een extra filter-stap.

psql
SELECT c.name, recent.*
FROM customers c
CROSS JOIN LATERAL (
    SELECT id, created_at FROM orders
    WHERE orders.customer_id = c.id
    ORDER BY created_at DESC LIMIT 3
) AS recent;

FILTER laat je meerdere voorwaardelijke aggregaten in één SELECT zetten zonder een woud van CASE WHEN:

psql
SELECT
    count(*) FILTER (WHERE status = 'paid')      AS betaald,
    count(*) FILTER (WHERE status = 'cancelled')  AS geannuleerd
FROM orders;
06

Constraints die verder gaan

CHECK en foreign keys ken je al. Dit hoofdstuk gaat over constraints die een hele categorie applicatiebugs elimineren door de database de regel te laten afdwingen in plaats van je eigen code.

vs MariaDB

Er bestaat geen MariaDB-equivalent voor exclusion- of temporal constraints. Het gangbare patroon daar is: applicatielogica die controleert op overlap vóór de INSERT, met alle race-condition-risico's van dien onder gelijktijdige schrijvers.

▸Het probleem: overlappende periodes

Stel: een werkplaats-reservering mag nooit overlappen met een bestaande reservering voor diezelfde ruimte. Vóór PostgreSQL 18 loste je dat op met een exclusion constraint op een GiST-index:

psql
CREATE EXTENSION IF NOT EXISTS btree_gist;

CREATE TABLE bookings (
    room_id integer,
    during  tsrange,
    EXCLUDE USING gist (room_id WITH =, during WITH &&)
);

Sinds PostgreSQL 18 kan hetzelfde declaratief via WITHOUT OVERLAPS, leesbaarder en rechtstreeks op de primaire/unieke sleutel:

psql
CREATE EXTENSION IF NOT EXISTS btree_gist;

CREATE TABLE bookings (
    room_id integer,
    during  tsrange NOT NULL,
    guest   text,
    PRIMARY KEY (room_id, during WITHOUT OVERLAPS)
);
-- functioneel identiek aan de EXCLUDE-versie hierboven,
-- maar leesbaar als "wat" in plaats van "hoe"

▸Temporal foreign keys: PERIOD

Ook referentiële integriteit kan tijdsgebonden: een PERIOD-foreign key controleert dat de periode van de rij volledig binnen de periode van de referentie valt, niet enkel dat de sleutel bestaat.

psql
CREATE TABLE room_capacity (
    room_id integer,
    available_period tsrange NOT NULL,
    PRIMARY KEY (room_id, available_period WITHOUT OVERLAPS)
);

CREATE TABLE bookings (
    room_id integer,
    booked_period tsrange NOT NULL,
    PERIOD FOREIGN KEY (room_id, booked_period)
        REFERENCES room_capacity (room_id, available_period)
);
-- een boeking kan enkel bestaan binnen een periode waarin de ruimte
-- effectief beschikbaar was gedeclareerd
Dieper: domains en deferrable constraints

Een domain is een hernoembaar, herbruikbaar type mét ingebouwde validatie: CREATE DOMAIN email AS text CHECK (VALUE ~ '^[^@]+@[^@]+$'); — daarna gebruik je email als kolomtype op elke tabel, en de check komt automatisch mee.

Deferrable constraints laten je een constraint pas laten controleren bij COMMIT in plaats van bij elke statement — nodig zodra twee rijen elkaar over en weer refereren en je ze in dezelfde transactie moet invoegen.

07

LISTEN/NOTIFY: ingebouwde pub/sub

PostgreSQL kan zelf berichten publiceren naar wie luistert — zonder message broker, zonder polling.

psql
-- Sessie A: luistert
LISTEN order_events;

-- Sessie B: publiceert (bv. vanuit een trigger na INSERT)
NOTIFY order_events, '{"order_id": 42, "status": "paid"}';

-- of automatisch via een trigger op de tabel zelf
CREATE FUNCTION notify_order_change() RETURNS trigger AS $$
BEGIN
    PERFORM pg_notify('order_events', row_to_json(NEW)::text);
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER orders_notify AFTER INSERT OR UPDATE ON orders
FOR EACH ROW EXECUTE FUNCTION notify_order_change();

In een FastAPI-app kan je met asyncpg een permanente connectie openhouden die op deze events luistert, en die vervolgens doorzet naar WebSocket-clients — of gebruikt om een Redis-cache gericht ongeldig te maken (meer daarover in hoofdstuk 11).

listener.py
import asyncpg, asyncio, json

async def listen():
    conn = await asyncpg.connect(DSN)

    def handler(connection, pid, channel, payload):
        event = json.loads(payload)
        print(f"order {event['order_id']} -> {event['status']}")

    await conn.add_listener('order_events', handler)
    while True:
        await asyncio.sleep(3600)
⚠️

Belangrijke grens: NOTIFY-payloads zijn beperkt tot 8000 bytes en berichten worden niet gebufferd voor een luisteraar die offline is — dit is fire-and-forget, geen persistente queue. Voor iets dat gegarandeerd afgeleverd moet worden (zoals een betaalstatus die absoluut niet verloren mag gaan), blijft een echte messagequeue of een outbox-tabel de juiste keuze. Voor lichte, "vertel de UI dat er iets veranderd is"-events is het ideaal.

08

Partitionering & schaalbaarheid

Declaratieve partitionering (sinds PostgreSQL 10) splitst een logisch één tabel fysiek op in meerdere onderliggende tabellen, transparant voor je queries.

psql
CREATE TABLE events (
    id         bigserial,
    created_at timestamptz NOT NULL,
    payload    jsonb
) PARTITION BY RANGE (created_at);

CREATE TABLE events_2026_07 PARTITION OF events
    FOR VALUES FROM ('2026-07-01') TO ('2026-08-01');
CREATE TABLE events_2026_08 PARTITION OF events
    FOR VALUES FROM ('2026-08-01') TO ('2026-09-01');

-- Een query met een datumfilter raakt automatisch enkel de relevante partitie(s)
SELECT * FROM events WHERE created_at >= '2026-08-15';

Naast RANGE bestaat ook LIST (discrete waarden, bv. per land) en HASH (gelijkmatig verdelen zonder natuurlijke sleutel). Het voordeel is drieledig: oude partities kan je in één DROP TABLE wegdoen in plaats van een trage DELETE, de planner kan irrelevante partities volledig overslaan (partition pruning), en onderhoud zoals VACUUM loopt per partitie in plaats van op een monolithische tabel. De extensie pg_partman automatiseert het aanmaken en opruimen van tijdgebaseerde partities.

Replicatie & high availability

Kort, ter aanvulling — dit bouwt verder op wat je al aan het verkennen bent met Patroni en pg_auto_failover.

Streaming (fysiek)Logisch
Wat repliceertelke WAL-byte, hele clusterrij-wijzigingen, per tabel/publicatie
Doel-schemaidentiek, byte-voor-bytemag afwijken (andere kolommen, andere indexen)
Cross-versieneeja
Typisch gebruikt voorhot standby / failoverselectieve sync, zero-downtime upgrades, ETL naar een analytics-DB

PostgreSQL heeft zelf geen ingebouwde automatische failover — vandaar de externe orchestrators die je al aan het bekijken bent: Patroni (gebruikt een distributed consensus store zoals etcd om een leider te verkiezen) en het lichtere pg_auto_failover. Vergelijk dat met MariaDB Galera Cluster: die biedt synchrone multi-master replicatie kant-en-klaar, maar met een ander consistentiemodel en andere operationele afwegingen (write-amplification bij elke node, gevoeliger voor netwerklatentie tussen nodes).

PostgreSQL + Redis: succesvol combineren

De kernregel: PostgreSQL is de bron van waarheid (ACID, durable, het enige dat écht niet verloren mag gaan), Redis is de snelheidslaag (in-memory, vluchtig, mag in het slechtste geval leeg beginnen zonder dataverlies).

▸Cache-aside: het standaardpatroon

main.py
async def get_balance(card_id: str) -> int:
    cached = await redis.get(f"balance:{card_id}")
    if cached is not None:
        return int(cached)

    row = await db.fetchrow(
        "SELECT balance FROM cards WHERE id = $1", card_id
    )
    await redis.set(f"balance:{card_id}", row["balance"], ex=30)  # TTL in seconden
    return row["balance"]

▸Invalidatie: het echte probleem

Cachen is eenvoudig; weten wanneer een cache-entry fout is geworden is het lastige deel. Drie strategieën, oplopend in complexiteit:

  • TTL alleen — simpelst, accepteert dat data tot X seconden verouderd kan zijn. Voeg jitter toe aan de TTL (bv. 30s ± 5s willekeurig) zodat niet honderden keys tegelijk verlopen en een cache stampede veroorzaken op je database.
  • Expliciete invalidatie bij schrijven — na een UPDATE in dezelfde applicatiecode ook DEL op de bijbehorende Redis-key. Simpel, maar elke schrijfpad in je code moet er consequent aan denken.
  • LISTEN/NOTIFY-gedreven invalidatie — een achtergrondtaak luistert op databasewijzigingen (hoofdstuk 07) en verwijdert de betrokken Redis-key, ongeacht via welk pad de wijziging binnenkwam (ook een handmatige psql-update door een beheerder). Robuuster, iets meer bewegende delen.
⚠️

Wat niet in Redis hoort: de enige kopie van geld-gerelateerde of anderszins onvervangbare data. Redis' persistentie (RDB-snapshots, AOF) is voor herstel na een crash, geen vervanging voor ACID-transacties. Gebruik Redis voor sessies, rate-limiting-tellers, korte-termijn caches en pub/sub — laat PostgreSQL beslissen over alles waarbij een verloren of dubbel verwerkte schrijfoperatie een probleem is.

Dieper: connection pooling met beide erbij

Een async FastAPI-app met meerdere workers kan makkelijk meer PostgreSQL-connecties openen dan de server aankan (elke connectie kost geheugen server-side, ongeveer 5-10 MB). Twee lagen om dat te beheersen: asyncpg.create_pool() of SQLAlchemy's async engine-pool binnen elke worker, én — bij veel workers of veel korte connecties — PgBouncer ervoor in transaction pooling-modus. Let op: transaction pooling combineert slecht met server-side prepared statements en met sessie-gebonden features zoals LISTEN — voor een permanente LISTEN-connectie gebruik je een aparte, niet-gepoolde verbinding. Redis-side gebruik je een vergelijkbare pool via redis.asyncio.ConnectionPool, met een eigen max-connections-limiet.

Voor een nieuwe async-first FastAPI-app is asyncpg (rechtstreeks, of via SQLAlchemy's async engine) meestal de snellere keuze puur op ruwe doorvoersnelheid; psycopg 3 wint als je ook synchrone code of exotischere libpq-features nodig hebt.

database.py
import asyncpg

pool: asyncpg.Pool | None = None

async def init_pool():
    global pool
    pool = await asyncpg.create_pool(
        dsn=DATABASE_URL,
        min_size=5,
        max_size=20,       # afgestemd op PgBouncer/serverlimiet, niet willekeurig hoog
        command_timeout=10,
    )

async def get_order(order_id: str):
    async with pool.acquire() as conn:
        return await conn.fetchrow(
            "SELECT id, payload FROM orders WHERE id = $1", order_id
        )

▸JSONB heen en terug met Pydantic

asyncpg geeft jsonb-kolommen terug als Python-str tenzij je een codec registreert; met een codec krijg je meteen dict/list, direct bruikbaar in een Pydantic-model.

database.py
import json

async def init_pool():
    global pool
    pool = await asyncpg.create_pool(dsn=DATABASE_URL, init=setup_codecs)

async def setup_codecs(conn):
    await conn.set_type_codec(
        'jsonb',
        encoder=json.dumps,
        decoder=json.loads,
        schema='pg_catalog',
    )
